## Test Philosophy (Established Mid-Session)

**Rule:** Every step must be tested the way a real user would execute it — via UI or API with proper auth/origin headers. No DB direct for user-facing flows.

**Why:** This is a pre-release validation for company use. If a UI flow is broken, we must find it here. DB shortcuts hide bugs that real users will hit.

**Acceptable DB direct:** Seeding test infrastructure that has no UI (e.g., first-time role/capability seeds). NOT acceptable for: creating profiles, controlled documents, routes, or approving anything a user would approve.

### FIX-08: `getActiveDocument` returns empty contentHash for new docs (F-10)
- **File:** `internal/modules/registry/delivery/http/routes.go:97`
- **Problem:** `COALESCE(d.content_hash_at_submit, '')` returns `""` for new docs (column is NULL before first submission) → UI sends `""` → `contracts/submit.go:validateSHA256Hex` rejects with `"content_hash is required"`.
- **Root cause analysis:** `content_hash` in submit is NOT an OCC guard — server wraps client hash in `ContentFormData` and recomputes its own hash deterministically. Validation only requires a valid 64-char SHA256 hex string. Initial document revision already has valid hash in `document_revisions` table.
- **Fix:** Updated query to fall back to latest revision hash:
  ```sql
  COALESCE(d.content_hash_at_submit,
           (SELECT r.content_hash FROM document_revisions r
             WHERE r.document_id = d.id
             ORDER BY r.created_at DESC LIMIT 1),
           '')
  ```
- **API rebuilt:** `go build ./apps/api/cmd/metaldocs-api` → BUILD OK.

## Known Recurring Gotchas

### ISO Segregation (Templates + Documents)
Same user cannot submit AND approve. `CheckSegregation("approver")` in `domain/approval.go` enforces `actorID != authorID`. **This is correct business behavior — not a bug.**

**Always use two users for approval flows:**
1. Login as second user: create `e2e-approver` via `POST /api/v1/iam/users`, give approver role in `general` area, login, approve via UI.
2. DB direct: only acceptable for test infrastructure setup (not recommended for approval flows in full E2E).

### Origin Protection
API rejects requests without correct browser Origin header (`AUTH_INVALID_ORIGIN`). Curl from terminal doesn't work for most endpoints.

**Workarounds:**
1. Browser DevTools console: `fetch('/api/v2/...', { method: 'POST', headers: {'Content-Type':'application/json'}, body: JSON.stringify({...}) })`
2. DB direct (only for infrastructure that has no user-facing path)

### React `_valueTracker` reset
Programmatic input `.value =` doesn't trigger React `onChange`. Requires:
```js
const inp = document.querySelector('input[name="controlled-document-code"]');
const nativeInputValueSetter = Object.getOwnPropertyDescriptor(window.HTMLInputElement.prototype, 'value').set;
const tracker = inp._valueTracker;
if (tracker) tracker.setValue('');
nativeInputValueSetter.call(inp, 'E2E-FULL-01');
inp.dispatchEvent(new Event('input', { bubbles: true }));
```

### documents-v2 routing
App uses state-based routing (`docsRoute` state + `window.history.pushState`). Direct URL navigation to `/documents-v2/{id}` doesn't work — must enter via workspace sidebar button first.
