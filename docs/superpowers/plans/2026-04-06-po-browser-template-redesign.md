# PO Browser Template Redesign Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the plain, unstyled `po-default-browser` template with a professional, enterprise-branded document using a global document design system that all future templates will reuse.

**Architecture:** Three deliverables: (1) `document-content.css` — a shared CSS design system imported into the browser editor, styling document primitives with the vinho palette; (2) `DocumentEditorHeader.tsx` — a React component that renders the locked document identity block above the editor from bundle metadata; (3) rewritten `po-default-browser` template HTML using the shared primitives, with a free-form Section 5 (Etapas).

**Tech Stack:** React + TypeScript (frontend), CSS Modules, Go (backend domain + tests), PostgreSQL migration SQL, CKEditor5 restricted editing.

## Design Decisions

These decisions are explicit and intentional — not gaps in the plan.

### 1. Template version reuse vs. new version
The plan rewrites `po-default-browser` version 1 in place (via migration 0058) rather than seeding a new version 2. This is safe because: (a) documents store their own `body` in the drafts/content table — the template body is only used to initialize new documents, not referenced by existing ones; (b) version 1 was seeded in development and has never been released to production users; (c) the existing `BrowserEditorBundle` returns the stored document `body`, not the live template body. If the project later reaches production and has released template snapshots to real users, new redesigns should bump the template version number. For this pre-production cycle, in-place update is correct.

### 2. Browser editor header vs. export path
The `DocumentEditorHeader` is a React-only UI layer rendered above the CKEditor surface. It does NOT get injected into the CKEditor `body` and is NOT stored with the document. This is intentional: DOCX export goes through the Go docgen layer, which auto-generates section 1 (Identificação) from database document metadata — the same data the React header reads. These are two separate rendering paths for the same source of truth. The browser header is a live editing aid; the export header is authoritative and generated from the DB. They are not expected to be identical in markup. This matches the design spec: "Sections 1 and 10 auto-generated by docgen on export, not in the browser template body."

### 3. Etapas schema contract decision
The PO v3 schema's `etapas` section defines structured `ItemFields` (Número, Título, Responsável, Prazo/SLA, etc.). These fields are used by: the Canvas editor (schema-driven fields), and docgen on DOCX export. The browser template deliberately uses free-form HTML for etapas — one model block, user duplicates freely, no enforced field labels. This means the browser template and the canonical PO v3 schema are intentionally misaligned on this section. The test skip in `TestPOBrowserTemplateCoversPOv3Schema` reflects an explicit design decision, not a hidden bug. If docgen export of etapas content is later required, that work belongs to the docgen layer reading saved document body HTML, not to the template schema.

## Task 3: Create `DocumentEditorHeader.tsx`

**Files:**
- Create: `frontend/apps/web/src/features/documents/browser-editor/DocumentEditorHeader.tsx`

Pure presentational component. Reads from `DocumentBrowserEditorBundleResponse` (already available in `BrowserDocumentEditorView`). No state, no side effects.

`DocumentListItem` does NOT have `createdAt` — use `versions[last].createdAt` (type `string`, ISO 8601). `ownerId` is the user's ID string; display name resolution is a future enhancement.

- [ ] **Step 1: Create the file**

```tsx
import type { DocumentBrowserEditorBundleResponse } from "../../../lib.types";
import styles from "./DocumentEditorHeader.module.css";

type DocumentEditorHeaderProps = {
  bundle: DocumentBrowserEditorBundleResponse;
};

export function DocumentEditorHeader({ bundle }: DocumentEditorHeaderProps) {
  const { document, versions } = bundle;
  const latest = versions.length > 0 ? versions[versions.length - 1] : null;
  const revision = latest ? String(latest.version).padStart(2, "0") : "—";
  const createdAt = latest?.createdAt ? formatDate(latest.createdAt) : "—";

  return (
    <div className={styles.header} data-testid="document-editor-header">
      <div className={styles.top}>
        <span>Metal Nobre</span>
        <span className={styles.code}>
          {document.documentCode ?? "—"} · Rev. {revision}
        </span>
      </div>
      <p className={styles.title}>{document.title}</p>
      <div className={styles.meta}>
        <MetaItem label="Tipo" value={document.documentType} />
        <MetaItem label="Elaborado por" value={document.ownerId} />
        <MetaItem label="Data" value={createdAt} />
        <MetaItem label="Status" value={document.status} />
        <MetaItem label="Aprovado por" value="—" />
      </div>
    </div>
  );
}

function MetaItem({ label, value }: { label: string; value: string }) {
  return (
    <span className={styles.metaItem}>
      <span className={styles.metaLabel}>{label}</span>
      <span className={styles.metaValue}>{value}</span>
    </span>
  );
}

function formatDate(iso: string): string {
  try {
    return new Date(iso).toLocaleDateString("pt-BR", {
      day: "2-digit",
      month: "2-digit",
      year: "numeric",
    });
  } catch {
    return "—";
  }
}
```

## Known Caveats (Codex Round 2 — delivered at max rounds)

These two issues were flagged by Codex but require scope decisions beyond this plan's boundary. They are documented here for the implementer to make an explicit call.

### Caveat A: Migration 0057 mutation
This plan modifies `migrations/0057_seed_po_browser_template.sql` to keep it in sync with `template.go` (required by `TestPOBrowserTemplateGoSQLParity`). Codex flagged this as rewriting migration history.

**Alternative (migration-additive approach):** Keep 0057 immutable. Move the full redesigned body only into `migrations/0058_update_po_browser_template.sql`. Change `TestPOBrowserTemplateGoSQLParity` to read 0058 instead of 0057 for the comparison, since 0058 becomes the "current state of truth" SQL. This is cleaner migration discipline.

**Current plan's rationale:** 0057 was a development seed, never applied to production. The parity test is a consistency guard — it must read whichever migration file is the canonical seed. If you choose the additive approach, update Task 6 to skip 0057 editing, update Task 7 to put the full template HTML in 0058, and update Task 8 to change the parity test to read `0058_update_po_browser_template.sql`.

### Caveat B: ownerId as display name
`DocumentEditorHeader` renders `document.ownerId` (an internal ID string) in the "Elaborado por" field. The design spec acknowledges this: "Elaborado por | frontend auth state (current user display name) | OwnerID is in the bundle; name resolved from auth context."

**To resolve:** Either (1) add `ownerDisplayName string` to the `BrowserEditorBundle` Go struct and populate it from the user directory before returning the bundle — this is the cleanest approach; or (2) use a frontend user-directory hook/context if one exists to resolve the ID to a name at render time. Both require scope beyond this plan. Until resolved, the header will show the raw user ID. This is acceptable for the current development phase.
