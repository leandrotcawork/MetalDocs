## Phase 10 followups
- **Placeholder delimiter deviates from plan** (`apps/docgen-v2/src/render/fanout.ts:58-65` and plan line 2250): plan text says SDT `placeholder:doc_code` (Word Structured Document Tag syntax), but implementation relies on docxtemplater's default `{doc_code}` delimiter via `processTemplateDetailed`. Tests and existing `processDocx.ts` use the same `{var}` style, so this is consistent internally. Fix: update plan to document `{var}` as the delimiter of record for Spec 3, or — if SDT is required for final templates — add a pre-pass that extracts text from `<w:sdt>` blocks into `{var}` tokens before handing to `processTemplateDetailed`. Flag at Phase 11 before client integration decides on template authoring contract.
- **`injectZones` prepends instead of replacing zone body** (`apps/docgen-v2/src/render/zoneInjection.ts:14-19`): current regex matches only `<w:bookmarkStart .../>` and inserts OOXML immediately after. Existing content between `zone-start:<id>` and `zone-end:<id>` is not cleared, so if a template ships with placeholder body content inside a zone (legitimate for WYSIWYG preview) the rendered DOCX will contain both the template content AND the injected OOXML. Matches plan line 2338 literal ("insert given OOXML immediately after"), but plan description at line 2336 says "between bookmarks", which most readers will interpret as replacement. Fix: decide semantics at Phase 11 entry. If replacement is intended, change regex to match `bookmarkStart ... bookmarkEnd` and substitute. If append is intended, document the contract in a top-of-file comment.
- **Silent drop of header/footer composition when template lacks well-known tags** (`fanout.ts:52-56`): `__header_composition__` and `__footer_composition__` are injected as placeholder values passed to `processTemplateDetailed`. If the template body has no matching `{__header_composition__}` / `{__footer_composition__}` tokens, the rendered OOXML is silently discarded with no warning or unreplaced-var entry (the vars are defined, so they're not "unreplaced"). Fix: either (a) add a contract check in `fanout()` that scans the template for these tags before composing sub-blocks and logs/errors if a non-empty composition has nowhere to go, or (b) document the template-authoring contract in a README for docgen-v2 noting these two required tags.
- **Service-token auth gates `/render/fanout` globally** (`apps/docgen-v2/src/service-auth.ts:9-16` + `src/index.ts:11`): `registerServiceAuth` runs an `onRequest` hook that rejects any URL except `/health` without a valid `x-service-token`. The new route inherits this; no route-level extra auth. This is intentional and matches other routes. Document in a top-of-file comment in `routes/fanout.ts` that this route is service-to-service only and trusts the token gate.
- **Test mock path relies on relative position** (`apps/docgen-v2/src/routes/__tests__/fanout.test.ts:39`): `vi.mock('../../s3', ...)` works only while the test stays at `src/routes/__tests__/`. If the test file moves, the mock silently stops applying and tests would hit real MinIO. Fix: use absolute alias or export a helper `mockS3()` from a shared test utility that resolves the path from the module under test's location.
- **`jszip` listed in `apps/docgen-v2/package.json` despite repo hoisting**: the dependency is declared explicitly at the app level (correct per workspace discipline), but node_modules resolution currently goes through the hoisted root copy on Windows pnpm. Confirm `pnpm -C apps/docgen-v2 install` still produces a matching version at publish/CI time, and add `jszip` to the CI integration test fixture for docgen-v2 so the app-level declaration is exercised independently of hoist layout.
- **`processTemplateDetailed` emits `Deprecated method ".setData"` stderr warnings**: visible in the test run from `@eigenpal/docx-js-editor@0.0.34`. Not introduced by Phase 10 (pre-existing in `processDocx.ts`) but now louder because fanout calls the same underlying API. Fix: upgrade docxtemplater usage in `@eigenpal/docx-js-editor` OR suppress the deprecation shim in vitest setup. Defer to eigenpal vendor task.

## Phase 9 followups
- **`WriteFreeze` missing `::uuid` casts on `tenant_id` / `id`** (`internal/modules/documents_v2/repository/snapshot_repository.go:79-86`): every other query in this file casts `$n::uuid` for `tenant_id` and `id` (see `WriteSnapshot` and `ReadSnapshot`). `WriteFreeze` passes them as raw text (`WHERE tenant_id=$3 AND id=$4`). Works today because pgx infers via the prepared statement, but diverges from the established pattern in this file and will fail under strict type-checking if the driver's inference changes. Fix: align with siblings — `WHERE tenant_id=$3::uuid AND id=$4::uuid`. Also worth adding a schema-aware `%s` / `r.table("documents")` is already used, so this is purely a cast fix.
- **`hex.DecodeString` error silently discarded** (`internal/modules/documents_v2/application/freeze_service.go:112`): `hashBytes, _ := hex.DecodeString(hashHex)` ignores error. Safe in practice because `ComputeValuesHash` always returns `hex.EncodeToString` output (64-char hex, never malformed), but the silent discard makes the invariant implicit. Fix: either (a) change `ComputeValuesHash` to return `[]byte` directly and have callers hex-encode when needed, or (b) keep string API but panic on malformed input since it signals a programmer error. Option (a) is cleaner since `WriteFreeze` consumes bytes anyway.
- **Stringification of computed resolver values uses `fmt.Sprintf("%v", rv.Value)`** (`freeze_service.go:91`): for primitive values (`string`, `int`, `time.Time`-formatted-by-resolver) this works, but for any complex type (slice, map, struct — e.g. `approvers []ApproverInfo`) it produces Go-syntax output like `[{alice 2026-04-23} {bob 2026-04-23}]` which is not a user-displayable string and definitely not DOCX-renderable. Resolvers in Phase 7 already return `any` but real use cases (approvers, revision_history) emit slices/structs. Fix: define a resolver-side contract that `Value` is always a string (or JSON-serializable primitive), and enforce it in `freeze_service` with a type-switch that rejects non-stringifiable types with `ErrValidationFailed`. Alternative: have resolvers return both `Value any` (for typed consumption later) and `DisplayText string` (for freeze).
- **No concurrency/idempotency guard on `Freeze()`** (`freeze_service.go:44`): parallel `Freeze()` calls on the same revision race on (a) computed placeholder upserts (last-writer-wins, acceptable) and (b) `WriteFreeze` finalize (last-writer-wins on `values_hash` + `values_frozen_at`, potentially causing audit divergence if two transactions computed different `values_hash` due to reading different user-placeholder states). Not a correctness issue yet since Phase 11 Submit should gate Freeze behind a state transition, but if callers invoke Freeze directly without state-machine enforcement, document that in the service godoc. Confirm alignment with plan §Freeze Model idempotency semantics.
- **No state guard on `WriteFreeze`** (`snapshot_repository.go:79-86`): the UPDATE has no `status` predicate, so it can silently overwrite a previously frozen revision (e.g. after a revision reopens). Plan at lines 2218-2227 specifies exactly this SQL, so leaving as-is. If Spec 3 adds an "unfreeze then re-freeze" flow (currently forbidden), add `WHERE values_frozen_at IS NULL` and surface `ErrAlreadyFrozen` when `RowsAffected() == 0`.
- **`byID[p.ID]` rebuild drops struct fields** (`freeze_service.go:101`): after the computed upsert, `byID[p.ID] = repository.PlaceholderValue{ValueText: &strVal}` zeros every other field of the struct (Source, ComputedFrom, ResolverVersion, InputsHash). Only `ValueText` is read later for the hash, so behavior is correct, but the zero-assignment is a footgun if future code reads other fields from the map. Fix: populate the full struct (mirror the upsert), or add a comment noting that only `ValueText` matters here.

## Phase 8 followups
- **XML escaping gap for user-provided text interpolation**: all five v1 sub-block renderers (`apps/docgen-v2/src/render/subblocks/doc_header_standard.ts`, `revision_box.ts`, `approval_signatures_block.ts`, `footer_controlled_copy_notice.ts`) concatenate values from `ctx.values` / `ctx.params` directly into OOXML strings via `${text}` inside `<w:t>…</w:t>`. Inputs such as `title`, `doc_code`, `display_name`, `description`, and `notice_text` can legally contain `<`, `>`, `&`, `"`, or the sequence `]]>` and will produce malformed XML or enable attribute/element injection at downstream assembly. The Phase 8 plan is silent on escaping at this layer (freeze/fanout phases handle final XML assembly). Fix: add a small `escapeXmlText(s string)` helper at `apps/docgen-v2/src/render/subblocks/xml.ts` that replaces `&` → `&amp;`, `<` → `&lt;`, `>` → `&gt;` (and `"` → `&quot;` for attribute contexts if later needed), then wire it through the shared `cell(text)` helpers in each sub-block. Add one unit test per sub-block asserting that a value containing `<` round-trips as `&lt;`. Consider lifting `str()` + `cell()` into that shared module while there (currently duplicated across `doc_header_standard.ts`, `revision_box.ts`, `approval_signatures_block.ts`).
- **Duplicated `str()` + `cell()` + `row()` helpers across sub-blocks**: `doc_header_standard.ts`, `revision_box.ts`, and `approval_signatures_block.ts` each redefine the same `str`/`cell`/`row` helpers. Low-cost DRY: extract into `apps/docgen-v2/src/render/subblocks/xml.ts` alongside the escape helper above. Defer until a fourth consumer (freeze/fanout) needs the same primitives to avoid premature abstraction.
- **`FooterPageNumbers` hardcodes fallback text `1` / `1`**: the `<w:t>1</w:t>` inside the `<w:fldSimple>` elements is the static fallback rendered by Word if it cannot evaluate the field. Acceptable per plan example verbatim, but worth verifying against final Word rendering in Phase 11 DOCX acceptance that the field actually updates (not that `1` is shown literally).
- **`approvers` / `revision_history` shape not validated**: sub-blocks accept any object in the arrays and silently produce empty cells for missing properties. Phase 9/10 freeze should validate shape at the snapshot boundary; sub-blocks should not grow defensive branching. Leave as-is; note for freeze contract.

- **Authz coverage**: `requireDocEditDraft` wired in `FillInService.SetPlaceholderValue`/`SetZoneContent`. Add integration test that seeds `user_process_areas` with a role lacking `doc.edit_draft`, calls service, expects denial. Follow pattern of approval-module integration tests.
