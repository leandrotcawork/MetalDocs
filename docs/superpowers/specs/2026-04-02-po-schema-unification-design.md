# PO Schema Unification Design

**Date:** 2026-04-02
**Status:** Approved
**Scope:** PO (Procedimento Operacional) document type schema — single source of truth

---

## Problem

The MetalDocs content builder reads document schemas from `document_profile_schema_versions.content_schema_json`, while the docgen engine reads from `document_type_schema_versions.schema_json`. Two separate schema definitions for the same document type leads to drift and maintenance burden. The old profile schemas contained legacy Carbone-era field structures that no longer match the new docgen architecture.

## Decision

**Approach B — Single source of truth via type schemas.** The schema lives exclusively in `document_type_schema_versions`. The content builder continues to call the existing profile schemas API, but the service layer resolves the content schema from the type schema table instead.

## 2. Schema Unification Wiring

### Current flow
```
ContentBuilderView
  → api.listDocumentProfileSchemas(profileCode)
  → service.ListDocumentProfileSchemas(profileCode)
  → repo: SELECT content_schema_json FROM document_profile_schema_versions
  → returns contentSchema to frontend
```

### New flow
```
ContentBuilderView
  → api.listDocumentProfileSchemas(profileCode)      ← NO CHANGE
  → service.ListDocumentProfileSchemas(profileCode)   ← CHANGED
  → repo: SELECT schema_json FROM document_type_schema_versions
          WHERE type_key = profileCode AND version = active_version
  → returns contentSchema to frontend                  ← SAME SHAPE
```

### Service layer change

In `ListDocumentProfileSchemas` (file: `internal/modules/documents/application/service_registry.go`):

After fetching profile schema versions from the repo, resolve `ContentSchema` from the type schema:

1. Look up `document_types` where `type_key = profileCode`
2. If found, fetch `schema_json` from `document_type_schema_versions` for the `active_version`
3. Set `item.ContentSchema = typeSchema.SchemaJSON`
4. If no matching type exists, fall back to the profile's own `content_schema_json` (backward compatibility)

### Profile ↔ Type mapping

Profile codes and type keys currently match 1:1 (`po`, `it`, `rg`). The service uses `profileCode` directly as `typeKey`. If they diverge in the future, add a `type_key` column to `document_profiles`.

## 3. Docgen Auto-Generated Sections

### Section 1 — Identificação (auto-generated)

Rendered by docgen from document metadata passed in the payload:

| Field | Source |
|-------|--------|
| `elaboradoPor` | `documents.owner_id` → user display name |
| `aprovadoPor` | Last approver from workflow/approvals → display name |
| `createdAt` | `documents.created_at` |
| `approvedAt` | Last approval date from workflow |

The Go backend populates these fields when building the `DocumentPayload` for docgen.

### Section 10 — Histórico de Revisões (auto-generated)

Rendered by docgen from version history:

```json
{
  "revisions": [
    { "versao": "1", "data": "2026-03-15", "descricao": "Criação do documento", "por": "Leandro" },
    { "versao": "2", "data": "2026-04-01", "descricao": "Atualização do fluxo", "por": "Leandro" }
  ]
}
```

Built from `document_versions` table: `version_number`, `created_at`, `change_summary`, and the user who created each version.

Docgen renders Section 1 before the schema-driven sections and Section 10 after them, using the existing styled table layout.

## 4. Migration Strategy

### Migration 0053: Clean slate + PO v3 schema

**Step 1 — Delete all test documents** (respecting FK order):

```sql
DELETE FROM metaldocs.document_collaboration_presence;
DELETE FROM metaldocs.document_edit_locks;
DELETE FROM metaldocs.document_attachments;
DELETE FROM metaldocs.document_versions;
DELETE FROM metaldocs.documents;
DELETE FROM metaldocs.outbox_events WHERE aggregate_type = 'document';
```

**Step 2 — Insert PO v3 schema:**

Insert into `document_type_schema_versions` with `type_key = 'po'`, `version = 3`, containing the full 8-section schema from Section 1 of this spec.

**Step 3 — Activate v3:**

```sql
UPDATE metaldocs.document_types SET active_version = 3 WHERE type_key = 'po';
```

Versions 1 and 2 remain in the table for history.

### Go defaults cleanup

`DefaultDocumentProfileSchemas()` in `domain/model.go` already returns empty `ContentSchema` (done in prior change). The `filterDefaultSchemas` fallback will return empty schemas, which is correct — the service layer now resolves from type schemas.

## 5. What Changes vs What Stays

### Changes
- `service_registry.go` — `ListDocumentProfileSchemas` resolves `ContentSchema` from type schemas
- `domain/model.go` — already cleaned (empty defaults)
- New migration `0053` — delete test docs, insert PO v3, activate
- `apps/docgen/src/generate.ts` — render auto-generated Sections 1 and 10 from metadata

### No changes
- Frontend `ContentBuilderView` — same API call, same response shape
- Frontend `DynamicEditor` — same schema format
- Frontend `toRuntimeDocumentSchema` — already handles all field types
- API handler response types — same `DocumentProfileSchemaResponse` shape
- `document_profile_schema_versions` table — still used for `metadata_rules_json`

---

## 6. Future Considerations

- **IT and RG schemas:** Same pattern — define in `document_type_schema_versions`, content builder picks them up automatically via the unified wiring.
- **Profile ↔ Type divergence:** If profile codes and type keys ever diverge, add `type_key` column to `document_profiles` table.
- **Schema versioning in editor:** When a new schema version is activated, existing documents retain their `profile_schema_version`. Content migration between schema versions is a separate concern.
