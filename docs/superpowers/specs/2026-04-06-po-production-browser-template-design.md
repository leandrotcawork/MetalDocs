# PO Production Browser Template Design

**Date:** 2026-04-06
**Status:** Approved
**Scope:** Production-ready browser editor template for PO (Procedimento Operacional) profile, covering all 8 editable sections of schema v3, with schema coverage test
**References:** `2026-04-06-document-authoring-v1-freeze.md`, `2026-04-04-template-assigned-browser-document-editor-design.md`

---

## 1. Objective

Replace the current smoke/pilot template (`po-default-canvas` v1, which covers only 2 of ~30 schema fields) with a production-ready browser editor template that covers all 8 editable sections of the PO v3 schema. This enables users to create real PO documents in MetalDocs.

---

## 2. Decisions Summary

| Decision | Choice |
|----------|--------|
| Editor | Browser / CKEditor5 (HTML with restricted-editing) |
| Profile scope | PO only (v1) |
| Complex fields | Hybrid: CKEditor tables for sections 7/8/9; rich editable blocks for etapas (section 5) |
| Placeholder text | Descriptive — guides the user on what to write |
| Visual identity | Formal institutional document |
| Header in editor | Simplified locked header (title + type); full header only on export |
| Section removal | Not supported — user writes N/A if section doesn't apply |
| Schema/template drift | Automated test validates template covers all schema labels |

---

## 3. Template Identity

| Attribute | Value |
|-----------|-------|
| `template_key` | `po-default-browser` |
| `version` | `1` |
| `profile_code` | `po` |
| `schema_version` | `3` |
| `editor` | `ckeditor5` |
| `content_format` | `html` |
| `definition_json` | `{}` (not used for browser path) |
| `body_html` | Full HTML (~200-300 lines) |

A new template key is used (not a new version of `po-default-canvas`) because the smoke template is a governed canvas (JSON Definition), and this is a browser editor (HTML body) — fundamentally different editing models.

## 5. HTML Markup Patterns

### 5.1 Simple fields (text/textarea) — sections 2, 3, 6

```html
<p><strong>Objetivo</strong></p>
<p><span class="restricted-editing-exception">Descreva o objetivo deste procedimento, incluindo o resultado esperado ao final da execução.</span></p>
```

Label locked in `<strong>`, editable zone in `<span class="restricted-editing-exception">` with descriptive placeholder.

### 5.2 Rich fields — sections 4, 5

```html
<p><strong>Descrição do processo</strong></p>
<div class="restricted-editing-exception">
  <p>Descreva o processo de forma detalhada, incluindo contexto e fluxo geral.</p>
</div>
```

`<div>` instead of `<span>` — allows paragraphs, lists, and rich formatting inside the editable zone.

### 5.3 Tables — sections 7, 8, 9

```html
<p><strong>Indicadores de Desempenho</strong></p>
<figure class="table restricted-editing-exception">
  <table>
    <thead>
      <tr><th>Indicador / KPI</th><th>Meta</th><th>Frequência</th></tr>
    </thead>
    <tbody>
      <tr><td>Ex: Taxa de retrabalho</td><td>Ex: &lt; 5%</td><td>Ex: Mensal</td></tr>
    </tbody>
  </table>
</figure>
```

Native CKEditor table with locked headers and one example row. User adds/removes rows freely.

### 5.4 Repeat (Etapas) — section 5

```html
<p><strong>Detalhamento das Etapas</strong></p>
<div class="restricted-editing-exception">
  <h3>Etapa 1</h3>
  <p><strong>Título:</strong> Descreva o título da etapa</p>
  <p><strong>Responsável:</strong> Cargo ou função responsável</p>
  <p><strong>Prazo / SLA:</strong> Prazo esperado</p>
  <p><strong>Descrição:</strong></p>
  <p>Descreva o que deve ser feito nesta etapa.</p>
  <p><strong>Observações:</strong> Informações complementares</p>
  <p><strong>Alertas / Desvios:</strong> Condições que exigem atenção especial</p>
  <hr>
  <p><em>Duplique o bloco acima para adicionar mais etapas.</em></p>
</div>
```

One rich editable block with a complete model step. Labels inside the editable zone (not locked) because the repeat section is free-form content — the user can reorganize, add fields, or remove what doesn't apply.

## 6. Versioning and Migration

### 6.1 Version model

```
po-default-canvas v1  (existing smoke, immutable)
po-default-browser v1 (new production template)
                  └── becomes the PO profile default
```

New key (`po-default-browser`) rather than new version of existing key because they represent fundamentally different editing models (governed canvas vs. browser HTML).

### 6.2 Migration artifacts

1. **New SQL migration** — inserts `po-default-browser` v1 into `document_template_versions` and updates `document_profile_template_defaults` to point PO → `po-default-browser` v1
2. **`DefaultDocumentTemplateVersions()`** — add new template entry in Go code
3. **`DefaultDocumentProfileTemplateDefaults()`** — PO default points to new key
4. **Frontend: remove `pilotTemplates.ts`** — frontend consumes template from backend via `browser-editor-bundle`, no more hardcoded template

### 6.3 Impact on existing documents

- Documents with `po-default-canvas` v1 snapshot: no impact (snapshot preserved in revision)
- New PO documents: receive `po-default-browser` v1 automatically
- Existing drafts: retain their current assignment; can be re-assigned manually
- Smoke template is not deleted — immutability guaranteed

## 7. Schema Coverage Test

### 7.1 Purpose

Automated Go test that validates the template HTML covers all fields defined in the PO v3 schema. Acts as a safety net against schema/template drift.

### 7.2 Mechanism

1. Load PO v3 schema (all 8 sections, all fields)
2. Load `po-default-browser` v1 template `body_html`
3. For each section: verify `section.Title` appears in the HTML
4. For each field: verify `field.Label` appears in the HTML
5. For table fields: verify each `column.Label` appears
6. For repeat fields: verify each `itemField.Label` appears
7. Fail with a clear message listing any missing labels

### 7.3 Scope limitations

The test validates **label presence only**. It does NOT validate:

- HTML structure correctness (CKEditor's responsibility)
- Editable zone placement
- Visual styling
- Semantic path mapping (no path extraction from HTML)

### 7.4 Test location

Same test file as existing template tests: `service_templates_test.go`

---

## 8. Out of Scope

- Templates for IT (Instrução de Trabalho) or RG (Registro) profiles
- Template admin/builder UI inside MetalDocs
- Section removal UX (user writes N/A)
- Schema v4 or schema changes
- Governed canvas template updates
- Collaborative editing
- Template visual builder

---

## 9. Acceptance Criteria

1. New template `po-default-browser` v1 exists in migrations and Go seed code
2. Template HTML covers all 8 editable sections (2–9) of PO v3 schema
3. PO profile default points to `po-default-browser` v1
4. Frontend `pilotTemplates.ts` is removed; template comes from backend bundle
5. Schema coverage test passes and catches missing labels
6. Existing documents with `po-default-canvas` v1 are unaffected
7. `go test ./...` passes
8. Frontend build passes
9. Browser editor smoke path works with the new template
